Key Insight: Cosentus is the RCM COMPANY, not the clinic

Cosentus serves 60+ clinics/practices as clients. The CLINIC does: patient registration, scheduling, encounters, clinical notes, AI Scribe dictation. Cosentus STAFF does: coding, billing, claim submission, payment posting, denials, AR follow-up, voice AI calls to payers. The Client Portal is how clinics upload work to Cosentus and track their revenue.

UAE vs US — What's Different

🇦🇪 UAE (me-central-1)

  • Patient ID: Emirates ID (784-YYYY-NNNNNNN-N)
  • Payers: TPAs — NAS, Mednet, NextCare, Almadallah, Daman, ADNIC, etc.
  • Regulator: DHA (Dubai Health Authority), HAAD (Abu Dhabi)
  • Claim format: DHA eClaim XML
  • Eligibility: RPA bot logs into TPA portal, scrapes
  • Prior Auth: DHA electronic pre-authorization
  • Currency: AED (د.إ)
  • Languages: Arabic + English
  • Compliance: NESA, DHA regulations, data residency
  • Coding extras: DHA activity codes alongside CPT/ICD
  • No SSN — Emirates ID is the universal identifier
  • Voice AI: Arabic + English, TPA IVR navigation

🇺🇸 US (us-west-1)

  • Patient ID: SSN (optional), MRN, DOB+Name
  • Payers: UHC, Aetna, BCBS, Cigna, Humana, Medicare, Medicaid, etc.
  • Regulator: CMS, state DOIs
  • Claim format: 837P (professional), 837I (institutional) EDI
  • Eligibility: EDI 270/271 real-time via clearinghouse
  • Prior Auth: Payer portal, phone, fax, X12 278
  • Currency: USD ($)
  • Languages: English + Spanish
  • Compliance: HIPAA, HITECH, state laws
  • Coding: CPT, ICD-10-CM, HCPCS Level II
  • No Emirates ID — SSN is optional, MRN is primary
  • Voice AI: English + Spanish, US payer IVR trees

Implementation Rule

The patient registration form MUST be region-aware. UAE shows Emirates ID field (required) and TPA dropdown. US shows SSN field (optional) and commercial payer dropdown. Never mix them on the same form. The region is set at the organization level (each client is configured as UAE or US).

Document Access — Every Module Needs Source Files

Core Principle: Documents Follow the Work, Not the Other Way Around

Staff should NEVER need to leave their module to find a document. Every module has a contextual document viewer — a side drawer or split pane that shows all files linked to the entity they're working on (patient, claim, encounter). The standalone "Documents" module is for managing, searching, classifying, and bulk operations — but document access is embedded everywhere.

Shared Document Viewer Component

Implementation: One Reusable DocViewer Component

Build ONE shared document viewer component that gets embedded in every module. It takes an entity ID (patient_id, claim_id, encounter_id) and fetches all linked documents. Features: PDF rendering, image zoom/rotate, side-by-side comparison, AI extraction overlay, download, print. Three display modes:

Split Pane

  • Used by: AI Coding, Denials, Payment Posting, AI Scribe
  • 50/50 or resizable split
  • Document on left, workspace on right
  • Can browse multiple docs (tabs)
  • Best for: modules where you READ a doc while DOING work

Side Drawer

  • Used by: Claims, A/R, Tasks, Credentialing, Watch & Track
  • Slides in from right (400-600px wide)
  • Overlays on top of the main view
  • Close to return to worklist
  • Best for: quick reference while working a list

Inline Preview

  • Used by: Eligibility, Scan & Submit, Patient profile
  • Thumbnail with click-to-expand
  • Small preview embedded in the form/card
  • Modal for full-size view
  • Best for: single doc reference (insurance card, ID)

Standalone "Documents" Module — What It's Actually For

The Documents module at /documents is NOT the primary way staff access docs. It's the MANAGEMENT layer:

  • → Document Vault: Search across ALL documents (full-text via OpenSearch)
  • → Bulk Upload: Upload batch of docs and AI auto-classifies them (superbill, EOB, clinical note, etc.)
  • → Fax Center: Inbound/outbound fax management
  • → Unlinked Docs: Queue of uploaded docs that haven't been linked to a patient/claim yet
  • → Retention: Compliance rules for document retention periods
  • → Audit Trail: Who accessed what document and when
Real-World Scenarios — What ACTUALLY Happens Day to Day

Why This Matters

Every module needs to handle the messy reality — not just the happy path. A patient no-shows. A payer takes back money they already paid. A doctor disagrees with AI codes. An ERA doesn't match any claim. The UI must support every one of these scenarios with clear status tracking, actions, and resolution workflows.

Scheduling — All Appointment States

  • Booked: Appointment created, patient confirmed
  • Confirmed: Patient confirmed via SMS/call/portal (automated reminder sent 24h + 2h before)
  • Checked In: Patient arrived at front desk
  • In Progress: Doctor has started the visit
  • Completed: Visit done, note pending
  • No-Show: Patient didn't come, didn't cancel. Track no-show count per patient. Auto-flag repeat no-showers. May trigger no-show fee. Slot wasted = lost revenue.
  • Cancelled (Patient): Patient cancelled ahead of time. Track cancellation reason. Slot freed for waitlist.
  • Cancelled (Provider): Doctor sick, emergency. All patients for that day need rescheduling. Batch notify patients.
  • Rescheduled: Moved to new date. Link old and new appointment for audit trail.
  • Walk-In: No appointment. Front desk creates on the spot. Marked as walk-in for reporting.
  • Waitlisted: No available slot. Patient on waitlist. Auto-notify when a cancellation frees up a slot.
  • Late Arrival: Patient came but past window. Doctor decides: see them (reduced time) or reschedule.
  • Telehealth: Virtual visit. Different workflow — no check-in, sends video link instead.
  • Recurring: Follow-up every 3 months. Auto-schedule series. Track compliance.

KPIs the scheduling module must track: No-show rate, cancellation rate, schedule utilization %, wait time (check-in to seen), avg visit duration, slots filled vs available.

Patient Registration — Edge Cases

  • New patient, complete info: Normal registration flow
  • New patient, missing insurance: Flag as "self-pay until verified". Allow partial registration.
  • Insurance changed: Patient got new job/plan. Old insurance goes to "prior coverage" history. New one becomes primary. All future claims use new payer.
  • Multiple insurances: Primary + Secondary + sometimes Tertiary. Coordination of Benefits (COB) rules apply. After primary pays, secondary gets billed for remainder.
  • Dependent (child): Patient is minor on parent's plan. Store subscriber info (parent) separately from patient info (child). Birthday rule for COB if both parents have coverage.
  • Self-pay / Uninsured: No insurance billed. Patient responsible for 100%. May qualify for sliding scale, charity care, or payment plan.
  • Workers' Compensation: Employer's WC carrier pays. Different claim form. Need employer info, injury date, WC case number.
  • Auto Accident / Liability: Third-party liability. Need accident date, attorney info, case number. May hold billing until settlement.
  • Patient deceased: Mark inactive. Close out any open claims. Balance goes to estate.
  • Demographic changes: Name change (marriage), address change, phone change. Audit trail on all changes.
  • Duplicate patient: Same person registered twice. Merge function needed. Carry over all claims, docs, history.

Eligibility Verification — All Outcomes

  • Active coverage, in-network: Green light. Show copay, deductible remaining, benefits.
  • Active but out-of-network: Warn. Higher patient responsibility. May need patient consent to proceed.
  • Inactive / Terminated: Red flag. Stop. Notify patient immediately. Ask for updated insurance or self-pay agreement.
  • Deductible not met: Patient owes more out-of-pocket. Show remaining deductible amount. Collect upfront if possible.
  • Prior auth required: Certain procedures need pre-approval. Flag before visit. Route to prior auth workflow.
  • Pending effective date: Coverage starts next month. Can't bill yet. Reschedule or collect self-pay.
  • COB conflict: Two active coverages. Determine primary vs secondary. Birthday rule, gender rule, or ask patient.
  • System down: TPA portal offline (UAE) or clearinghouse timeout (US). Queue for retry. Allow provisional visit with disclaimer.
  • Batch overnight check: Run all next-day appointments through eligibility overnight. Flag any issues for front desk to resolve before patient arrives.

AI Scribe — What Can Go Wrong

  • Normal: Dictate → AI generates SOAP → doctor signs → routes to coding ✓
  • Doctor disagrees with AI codes: Overrides suggestions. Override is tracked (AI learns from it).
  • Bad audio quality: Background noise, mumbling. AI flags low-confidence sections in yellow. Doctor must manually fill those sections.
  • Multi-language visit: Patient speaks Arabic, doctor explains in English. AI handles code-switching. Transcript shows both languages.
  • Doctor wants to amend signed note: Addendum workflow. Original note preserved. Amendment logged with timestamp and reason.
  • Incomplete note: Doctor gets interrupted mid-visit. Note saved as draft. Can resume later. Draft doesn't route to coding.
  • Multiple patients queued: Doctor finishes 3 visits before sitting down to dictate. Dictates for each patient sequentially. Each generates separate note.
  • Specialist referral generated: During visit, doctor decides to refer. AI detects referral language and auto-creates referral order. (Sprint 4+)
  • Sensitive content: Mental health, substance abuse. AI must respect 42 CFR Part 2 restrictions. Extra consent required for sharing.

Coding — Daily Scenarios

  • AI codes correct: Coder approves. Done in 30 seconds per chart.
  • AI codes wrong: Coder corrects. Override tracked. AI retrains.
  • Missing documentation: Clinical note doesn't support the code. Coder sends QUERY back to doctor: "Please clarify if this was a level 3 or 4 visit. Documentation shows..." Doctor responds via query queue.
  • Upcoding flag: AI or audit detects code higher than documentation supports. Compliance alert. Must review before submission.
  • Downcoding risk: AI or coder suggests lower code than documentation supports. Lost revenue. Flag for review.
  • Modifiers needed: Multiple procedures same session (modifier 51/59). Bilateral (modifier 50). Distinct procedure (modifier XE/XS). Reduced service (modifier 52). AI should suggest modifiers.
  • E&M level determination: 2021 guidelines (MDM-based). AI evaluates: number of problems, data reviewed, risk. Suggests E&M level (99211-99215).
  • Unlisted code: Procedure has no specific CPT. Use unlisted code + description + comparable code for pricing. Needs supporting documentation.
  • Bundling/unbundling: CCI edits. Some code pairs can't be billed together. AI checks CCI edits before coder even sees it. NCCI edits for Medicare.
  • Held for audit: Random or targeted chart pulled for internal audit before submission. Compliance requirement.

Claims Center — Every Claim State

  • Draft: Claim created but not yet scrubbed/submitted
  • Scrubbing: AI running payer rules. Checking for errors.
  • Scrub Failed: Errors found. Biller must fix: missing auth, invalid code, wrong payer ID, etc. Shows specific error list.
  • Ready to Submit: Passed all scrub rules. In submission queue.
  • Submitted: Sent to clearinghouse (US) or DHA (UAE). Awaiting acknowledgment.
  • Accepted (999/277): Clearinghouse or payer accepted for processing. Now waiting for adjudication.
  • Rejected: Clearinghouse rejected (technical error — bad format, missing field). Fix and resubmit. NOT a denial.
  • In Process: Payer is adjudicating. Typical: 14-45 days.
  • Paid (Full): Payer paid contracted amount. ERA received. Auto-posted.
  • Paid (Partial): Payer paid less than expected. Could be: applied to deductible, out-of-network reduction, bundling, downcoded. Review needed.
  • Denied: Payer refused payment. Denial reason code (CARC/RARC for US, TPA reason for UAE). Routes to Denials module.
  • Appealed: Denial appealed. Track appeal level (L1/L2/L3), deadline, supporting docs sent.
  • Corrected/Replaced: Original claim had error. Void original + submit corrected (837 frequency code 7 or 8).
  • Secondary Billing: Primary paid their portion. Remaining balance billed to secondary insurance with primary's EOB attached.
  • Patient Responsibility: After insurance, remaining balance → patient statement.
  • Write-Off: Balance deemed uncollectable. Contractual write-off (normal) vs bad debt write-off (needs approval).
  • Timely Filing Risk: Approaching payer's filing deadline. URGENT flag. Typical: 90 days (commercial), 365 days (Medicare), varies by payer.

Payment Posting — All Scenarios

  • ERA auto-match: AI matches payment to claim, posts adjustments. Human reviews exceptions only. Best case.
  • Orphan payment: Payment received but doesn't match any claim in system. Unmatched queue. Manual investigation needed.
  • Underpayment vs contract: Payer paid less than contracted rate. AI flags for Contract Manager review. May appeal.
  • Overpayment: Payer paid more than expected. Must refund. Compliance risk if not returned promptly. Track refund.
  • Recoupment / Takeback: Payer TAKES BACK money they already paid. Shows as negative on ERA. Often from audit. Must track and potentially appeal.
  • Patient copay at time of service: Collected at front desk. Post to patient account. Reconcile with ERA when insurance pays.
  • Patient payment (after insurance): Patient pays balance via statement, portal, phone. Post to account. Multiple payment methods: check, credit card, payment plan.
  • Adjustment codes: Each adjustment has a reason (CO = contractual obligation, PR = patient responsibility, OA = other adjustment, PI = payer initiated). AI must categorize correctly.
  • ERA with multiple claims: Single ERA file contains payments for 50+ claims. Batch posting. Exceptions flagged individually.
  • Manual EOB (no ERA): Smaller payers send paper EOB. Must be scanned, OCR'd, manually posted. Or uploaded and AI extracts payment info.
  • Write-off approval: Contractual write-offs auto-approved. Bad debt write-offs need supervisor approval above threshold (e.g., $500+).

Denials & Appeals — Reason Categories and Actions

  • No prior authorization: Appeal with: auth number if it exists, or request retroactive auth. Attach clinical necessity docs.
  • Not medically necessary: Appeal with: clinical notes, peer-reviewed literature, letter of medical necessity from doctor.
  • Timely filing: Appeal with: proof of original submission (clearinghouse report, fax confirmation, tracking number).
  • Duplicate claim: Prove it's not a duplicate. Show different dates of service, or different procedures, or show original was voided.
  • Patient not eligible: Verify coverage was active on DOS. If yes, appeal with eligibility proof. If no, rebill correct payer or bill patient.
  • Coding error (from payer perspective): Review their logic. If wrong, appeal. If valid, correct code and resubmit.
  • Missing information: Supply the missing info. Resubmit within payer's correction window.
  • L1 → L2 → L3 appeal escalation: Each level has its own deadline, format, and recipient. L3 is often external/independent review. Track where each appeal sits.
  • Pattern detection: Same denial reason hitting multiple claims → systemic issue. Example: "No auth" denials spiking → prior auth process is broken. Alert supervisor.
  • Partial overturn: Appeal partially successful. Payer pays some, but not all. May need L2 appeal for remainder.
  • Payer audit / Post-payment review: Payer audits past claims. May demand money back. Track audit, respond with records, contest if appropriate.

A/R Management — Prioritization and Actions

  • High-dollar aged account: $5,000+ sitting 90+ days. Priority 1. Voice AI calls payer. If no response, escalate to supervisor.
  • Approaching timely filing: URGENT. 30 days before deadline. Must get status or resubmit. Voice AI auto-scheduled call.
  • Payer says "in process" repeatedly: Call log shows 3 "in process" responses over 60 days. Escalation trigger. Request supervisor review or formal complaint.
  • Patient portion unpaid: After insurance paid, patient balance outstanding. Statement cycle: 30 days → 60 days → 90 days → final notice → collections consideration.
  • Payment plan: Patient can't pay full balance. Set up installment plan. Track monthly payments. Flag if payment missed.
  • Hardship / Charity care: Patient qualifies for financial assistance. Application workflow. Discount or write-off with documentation.
  • Bad debt transfer: Account uncollectable after exhausting internal efforts. Transfer to external collections agency. Requires director approval. Track agency performance.
  • Credit balance: Patient or payer overpaid. Account has negative balance. Must refund within regulatory timeframe (varies by state/payer). Compliance risk.
  • Shift handoff: AR rep worked accounts all day. Shift ends. Incomplete follow-ups need notes and handoff to next shift. Timezone-aware for UAE/US.

Voice AI — Call Scenarios

  • Outbound to payer (claim status): AI dials payer → navigates IVR → reaches rep → provides claim info → gets status → logs result → updates claim.

How Cosentus Validates Accuracy

The Clinic Sends Codes on Superbills — But We Don't Trust Them Blindly

Many clinics have superbills with pre-printed CPT codes that doctors tick off. These are a starting point, not the final word. Here's the validation chain:

  • 1. Client uploads superbill (with ticked codes) + clinical note
  • 2. AI Textract extracts ticked codes from superbill + reads clinical note
  • 3. AI Auto-Coding independently generates code suggestions from the clinical note text
  • 4. AI compares: Superbill codes vs AI-suggested codes. Matches are green. Mismatches are flagged yellow.
  • 5. Human coder reviews: Sees both sets side by side. Makes final decision. Common fixes: upcoding correction, missing modifier, bundling issue, documentation doesn't support code.
  • 6. AI Claim Scrubbing runs payer-specific rules on final codes before submission

This way: even if the doctor ticks the wrong box, or the AI misreads a checkbox, or the codes don't match documentation — a human coder catches it before it goes to the payer.
